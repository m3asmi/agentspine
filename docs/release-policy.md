# Release Policy

## Quality Gates

All code must pass before merge:

1. **Lint**: `uv run ruff check .` — zero violations
2. **Format**: `uv run ruff format --check .` — no changes needed
3. **Type check**: `uv run mypy .` — zero errors (strict mode)
4. **Tests**: `uv run pytest` — all pass
5. **Coverage**: `uv run pytest --cov` — minimum 80%

## Branch Strategy

- `main` — stable, always passes all gates
- `feature/*` — one branch per task (e.g., `feature/P0-002-platform-api`)
- `fix/*` — bug fixes
- `chore/*` — maintenance, docs, tooling

## Commit Format

```
<type>: <description>

<optional body>
```

Types: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `ci`

## Pull Request Rules

- One PR per task
- PR title matches task ID: `feat: P0-002 platform API interfaces`
- Must pass all quality gates
- Must include tests for new code
- Reviewed before merge (or self-reviewed for low-risk)

## Release Process

- Semantic versioning: `MAJOR.MINOR.PATCH`
- Phase completion triggers minor version bump
- Breaking changes trigger major version bump
- Tag releases: `v0.1.0`, `v0.2.0`, etc.

## Rollback

- Keep previous version deployable
- Database migrations must be reversible
- Feature flags for gradual rollout (when applicable)
