# Security Policy

## Mandatory Security Checks (OWASP Top 10)

Before any commit touching auth, permissions, or external APIs:

- [ ] No hardcoded secrets (API keys, passwords, tokens)
- [ ] All user inputs validated (Pydantic schemas)
- [ ] SQL injection prevention (SQLAlchemy parameterized queries only)
- [ ] XSS prevention (no raw HTML rendering)
- [ ] CSRF protection on state-changing endpoints
- [ ] Authentication verified on all non-public endpoints
- [ ] Authorization checked (deny-by-default)
- [ ] Rate limiting on all external-facing endpoints
- [ ] Error messages don't leak internal state or stack traces

## Secret Management

- **Never** hardcode secrets in source code
- Use `.env` files for local development (excluded from git)
- Use environment variables or a secret manager in production
- Validate required secrets are present at startup
- Rotate any secrets that may have been exposed
- `.env.example` contains only placeholder values

## Permission Model

- **Deny-by-default**: plugins must request permissions explicitly
- Permissions declared in manifest, granted by kernel at activation
- Runtime enforcement: capability invocations checked against granted permissions
- Four executor profiles: `read_only`, `workspace_safe`, `workspace_dev`, `bounded_live_write`

## Risk Tiers

| Tier | Scope | Approval Required |
|------|-------|-------------------|
| R0 | Read-only | Auto |
| R1 | Workspace generation | Auto |
| R2 | Bounded local write | Policy check |
| R3 | Live external action | Human approval |

## PII Handling

- No PII stored in logs
- PII in database must be identified and access-controlled
- Audit events must not contain raw PII (use references/hashes)

## Data Boundaries

- Plugin code may only access `platform_api` — no internal module imports
- Plugins cannot access other plugins' data directly
- All cross-plugin communication goes through registries

## Security Response Protocol

If a security issue is found:
1. STOP immediately
2. Fix CRITICAL issues before continuing
3. Rotate any exposed secrets
4. Review entire codebase for similar issues
5. Add regression test for the vulnerability
